# 品牌合规审核 - 速度优化方案

## 当前问题分析

### 1. 审核速度慢的原因

| 问题 | 当前状态 | 影响 |
|------|---------|------|
| 每张图片单独调用API | 10张图 = 10次API调用 | 每次都要传输完整的规范Prompt (~3000 tokens) |
| 图片未经压缩 | 最大4096px，quality=95 | 高分辨率图片消耗大量视觉Token |
| 无并发处理 | 串行逐张审核 | 无法利用并发能力 |
| 无效的缓存功能 | 内存缓存，但图片通常只审核一次 | 占用内存，误导用户 |
| 冗长的Prompt | 完整的20项检查规则 | 增加输入Token消耗 |

### 2. 时间估算修正

**实际测试数据：**
- 单张图片审核：约 45-90 秒（取决于图片复杂度和网络）
- 10张图片（当前串行）：约 8-15 分钟
- 10张图片（优化后并发）：约 2-4 分钟

---

## 优化方案

### 优化一：移除无效缓存功能

**理由：**
- 审核场景下，每张图片通常只审核一次
- 内存缓存占用资源但几乎无命中
- 误导用户以为有缓存加速

**实施：**
- 删除 `AuditCache` 类
- 删除 `llm_service` 中的缓存相关代码
- 更新界面提示信息

### 优化二：批量审核 - 并发请求

**核心思路：** 多个图片并发调用API，而非串行等待

**方案对比：**

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| 方案A：合并请求 | 一次请求审核多张图 | 节省Prompt Token | 上下文窗口限制，响应解析复杂 |
| 方案B：并发请求 | 同时发送多个独立请求 | 实现简单，可靠 | 每张图仍需传输Prompt |
| 方案C：混合方案 | 少量合并 + 并发 | 平衡效率与可靠性 | 实现复杂 |

**推荐：方案B（并发请求）**

原因：
1. 多模态模型上下文窗口有限（通常128K tokens）
2. 单张图片Token消耗大（约1000-4000 tokens）
3. 合并请求响应解析复杂，容错性差
4. 并发请求实现简单，效果明显

**并发数量计算：**
```python
# 根据模型限制动态调整
MAX_CONCURRENT = min(5, API_RATE_LIMIT)  # 默认最大5并发
```

### 优化三：图片智能压缩

**当前问题：**
- 最大尺寸4096px仍然过大
- quality=95几乎无损，但LLM不需要如此高清
- PNG格式体积大

**优化策略：**

| 参数 | 当前值 | 优化值 | Token节省 |
|------|--------|--------|-----------|
| 最大尺寸 | 4096px | 1920px | ~50% |
| 图片质量 | 95 | 75 | ~40% |
| 输出格式 | PNG/JPEG | JPEG | ~30% |

**智能压缩算法：**
```python
def smart_compress(image_data: bytes) -> tuple[str, bytes]:
    """智能压缩图片"""
    img = Image.open(BytesIO(image_data))

    # 1. 调整尺寸 - 降低到1080P级别
    max_dimension = 1920
    if max(img.size) > max_dimension:
        ratio = max_dimension / max(img.size)
        img = img.resize(...)

    # 2. 转换格式
    img = img.convert("RGB")

    # 3. 质量压缩
    buffer = BytesIO()
    img.save(buffer, format="JPEG", quality=75, optimize=True)

    # 4. 检查大小，如仍过大则进一步压缩
    if len(buffer.getvalue()) > 500_000:  # 500KB
        buffer = BytesIO()
        img.save(buffer, format="JPEG", quality=60, optimize=True)

    return "jpeg", buffer.getvalue()
```

**预估效果：**
- 原图 5MB → 压缩后 200-500KB
- Token消耗减少 60-80%
- 审核速度提升 30-50%

### 优化四：Prompt精简

**当前Prompt长度：** ~3000 字符（约1500 tokens）

**精简方案：**

| 部分 | 当前 | 精简后 | 节省 |
|------|------|--------|------|
| Logo规范 | 7条详细说明 | 核心要点 | 50% |
| 色彩规范 | 3条 | 合并为1条 | 66% |
| 字体规范 | 2条 | 保留 | 0% |
| 排版规范 | 4条 | 合并为2条 | 50% |
| 风格规范 | 4条详细说明 | 关键词列表 | 70% |
| 输出格式 | 完整JSON示例 | 精简格式 | 60% |

**精简后Prompt长度：** ~1200 字符（约600 tokens）

### 优化五：实施并发审核

**架构调整：**

```
当前流程（串行）:
图1 → API → 等待 → 图2 → API → 等待 → 图3 → API → 等待 → ...

优化后流程（并发）:
┌── 图1 → API ──┐
├── 图2 → API ──┤
├── 图3 → API ──┼→ 汇总结果
├── 图4 → API ──┤
└── 图5 → API ──┘
```
